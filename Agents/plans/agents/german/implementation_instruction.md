# Instruction For Execution Model: German Obsidian Refactor

## Task

Нужно реализовать рефакторинг немецкого навыка в проекте AntigravityAgents.

Цель: уйти от агрегированных файлов `vocabulary.md` и подобных к модели, где каждое слово и каждая фраза сохраняются как отдельная markdown-заметка в Obsidian-совместимой структуре.

## Read First

Перед началом прочитай эти файлы:

1. `core/utils_obsidian.py`
2. `core/orchestrator_v2.py`
3. `core/skills/german_teacher.py`
4. `plans/german_obsidian_refactor_plan.md`
5. `knowledge/german/_templates/word_note_template.md`
6. `knowledge/german/_templates/phrase_note_template.md`
7. `knowledge/german/_templates/learning_plan_template.md`

## Required Outcome

После реализации система должна:

1. Сохранять каждое немецкое слово в отдельный markdown-файл.
2. Сохранять каждую немецкую фразу в отдельный markdown-файл.
3. Вести учебный план в отдельном markdown-файле с учетом выполнения задач.
4. Корректно работать и в обычном режиме, и в режиме `obsidian_vault_simulation`.
5. Оставаться совместимой с текущим транспортным слоем.

## Mandatory Word Note Rules

Это уже согласовано с пользователем и не должно быть переосмыслено.

### Part Of Speech Detection

Система должна попытаться определить, является ли вход:

- существительным;
- глаголом;
- или неуверенно определенным словом общего типа.

Достаточно реализовать практичный эвристический подход. Не нужен сложный NLP-анализатор.

### File Naming

Если слово распознано как существительное:

- имя файла строится без артикля;
- пример: `die Wohnung -en` -> `wohnung.md`.

Если слово распознано как глагол:

- имя файла строится по инфинитиву;
- пример: `gehen, ging, ist gegangen` -> `gehen.md`.

Если тип не определен уверенно:

- использовать нормализованную базовую форму;
- сохранить файл как обычную word note;
- вернуть в статусе предупреждение о неуверенном определении типа.

### Word Frontmatter

Каждая заметка слова обязана содержать frontmatter с полями:

- `wort`
- `übersetzung`
- `Beispiel`
- `created`
- `tags`
- `status`
- `in_dict`

Фиксированные значения:

- `tags`: `Deutsch`, `Words`
- `status`: `nova`
- `in_dict`: `false`

Для существительного в `wort` нужно хранить форму с артиклем и множественным числом.

Пример:

```yaml
---
wort: die Wohnung -en
übersetzung: квартира
Beispiel:
  - Ich suche eine neue Wohnung in Berlin.
  - Die Wohnung ist klein, aber gemütlich.
created: 2026-03-10 21:15
tags:
  - Deutsch
  - Words
status: nova
in_dict: false
---
```

Для глагола в `wort` нужно хранить три формы.

Пример:

```yaml
---
wort: gehen, ging, ist gegangen
übersetzung: идти
Beispiel:
  - Ich gehe heute früh nach Hause.
  - Gestern ist er zu Fuß gegangen.
created: 2026-03-10 21:15
tags:
  - Deutsch
  - Words
status: nova
in_dict: false
---
```

## Phrase Note Rules

Для фразы допустима свободная форма, но должна быть отдельная заметка и вынесенный шаблон.

Минимум нужен такой frontmatter:

- `phrase`
- `übersetzung`
- `context`
- `created`
- `tags`
- `status`
- `in_dict`

Тело заметки должно содержать:

- `## Usage`
- `## Examples`
- `## Notes`

## Learning Plan Rules

Учебный план ведется отдельным файлом.

Нужно поддержать минимум такие операции:

1. создать план, если файла нет;
2. обновить цели и текущий фокус;
3. добавить задачу;
4. отметить задачу выполненной;
5. добавить запись в progress log.

Допустим рабочий свободный формат, но шаблон должен быть вынесен из кода.

## Suggested File Structure

Используй такую структуру:

```text
knowledge/
  german/
    words/
    phrases/
    grammar/
    plans/
    reviews/
    _templates/
```

## Implementation Requirements

### 1. Create A Storage Layer

Создай отдельный модуль, предпочтительно:

- `core/skills/german_storage.py`

Он должен отвечать за:

- выбор корня хранилища;
- VDS/Home mode detection;
- построение путей;
- нормализацию имен файлов;
- чтение шаблонов;
- запись новых заметок;
- безопасичное обновление существующих заметок.

### 2. Refactor German Skill API

Текущий немецкий навык должен перейти к операциям типа:

- `save_word(...)`
- `save_phrase(...)`
- `save_grammar_note(...)`
- `create_or_update_learning_plan(...)`
- `mark_plan_task_done(...)`
- `append_plan_progress(...)`

Допускается сохранить `save_knowledge(...)` как совместимый фасад, но внутренняя логика должна вести в новые операции.

### 3. Keep Transport Compatibility

Нельзя ломать существующую механику транспорта.

Это значит:

- если существует `obsidian_vault_simulation`, запись должна идти туда;
- структура должна оставаться файловой;
- не нужно менять export/import batch format без крайней необходимости.

## Existing Data Policy

Нельзя автоматически удалять или мигрировать старые агрегированные файлы деструктивно.

Допустимо:

- перестать использовать их как основное место записи;
- оставить как legacy;
- добавить явный комментарий в коде или статусе, что новая запись идет в поштучную модель.

## Update Strategy For Existing Notes

Если заметка уже существует:

1. не перезаписывай весь файл целиком без необходимости;
2. не уничтожай ручные правки пользователя;
3. обновляй frontmatter и стандартные секции аккуратно;
4. если безопасное обновление сложно, разрешено вернуть понятный статус вместо агрессивной перезаписи.

## What Not To Do

1. Не проектируй новую базу данных.
2. Не внедряй полноценный алгоритм интервального повторения на этом шаге.
3. Не меняй несвязанные части оркестратора.
4. Не меняй транспортный слой без прямой необходимости.
5. Не придумывай новый формат word note: он уже согласован.

## Validation

Нужно добавить проверяемый локальный тест или тестовый скрипт, который подтверждает:

1. создание существительного как отдельного файла без артикля в имени;
2. создание глагола как отдельного файла по инфинитиву;
3. создание phrase note как отдельного файла;
4. создание или обновление learning plan;
5. работу в `obsidian_vault_simulation`.

## Definition Of Done

Считать задачу завершенной можно только если:

1. новое слово создает отдельную markdown-заметку;
2. новое существительное создает файл без артикля в имени;
3. новый глагол создает файл по инфинитиву;
4. frontmatter word note соответствует согласованному контракту;
5. новая фраза создает отдельную markdown-заметку;
6. учебный план хранится отдельным файлом и умеет учитывать выполнение;
7. все это работает в VDS-compatible режиме через `obsidian_vault_simulation`.

## Output Expectations

В конце работы верни:

1. краткое описание, какие файлы были изменены;
2. какие новые методы добавлены;
3. как выполнялся тест;
4. какие ограничения остались;
5. есть ли открытые вопросы по неоднозначному определению части речи.
