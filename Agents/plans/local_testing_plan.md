# Подплан реализации системы автоматизированного локального тестирования (Local-First Testing)

Этот план описывает внедрение системы проверки работоспособности интерфейса и бэкенда на локальной машине перед деплоем на VDS.

## 1. Цель
Исключить ситуации, когда на сервер попадает нерабочий код, отсутствует секция в настройках или возникают ошибки 404.

## 2. Технологический стек
- **Инструмент:** `Playwright` (Python-библиотека для автоматизации браузеров).
- **Среда:** Локальный запуск FastAPI на порту 8080.
- **Отчетность:** Скриншоты состояния страниц и логи консоли браузера.

## 3. Этапы реализации

### Этап 1: Инфраструктура тестов
- Создание директории `Agents/tests/ui/`.
- Написание базового скрипта `test_ui_smoke.py`, который:
    1. Запускает локальный сервер (если не запущен).
    2. Открывает `http://localhost:8080/`.
    3. Проверяет статус ответа (не 404).
    4. Делает скриншот главной страницы.

### Этап 2: Проверка функционала Realtime
- Добавление тестов для страницы настроек:
    1. Переход на `/settings`.
    2. Проверка наличия вкладок агентов.
    3. Проверка наличия секции "Realtime Voice (Gemini Live)".
    4. Имитация сохранения настроек и проверка ответа API.

### Этап 3: Интеграция в рабочий процесс (Workflow)
- Создание скрипта-чекера `check_before_push.sh`.
- Скрипт должен:
    1. Выполнить `py_compile` для проверки синтаксиса.
    2. Запустить UI-тесты через Playwright.
    3. При обнаружении ошибок — блокировать возможность коммита/пуша.

### Этап 4: Визуальное подтверждение
- Настройка автоматического сохранения скриншотов в `Agents/tests/ui/screenshots/`.
- Я (Roo) буду анализировать эти скриншоты перед тем, как просить у вас разрешение на деплой.

## 4. Ожидаемый результат
- Гарантия 100% работоспособности интерфейса на момент пуша.
- Сокращение времени на "отладку через деплой" (когда мы правим ошибки уже на сервере).
- Возможность проверять сложные JS-сценарии (WebSocket соединение, работа микрофона в эмуляции).

## 5. Следующие шаги
1. Установить `playwright` в локальное виртуальное окружение.
2. Написать первый тест на проверку страницы `/settings`.
