# План реализации: поштучные заметки Obsidian для немецкого навыка

## Что означает "создается запись в файле"

В текущей реализации навык учителя немецкого языка не создает отдельную заметку на каждое слово или фразу. Он дописывает текст в агрегированные markdown-файлы, например:

- `vocabulary.md`
- `grammar.md`
- `progress.md`
- `learning_plan.md`

Это удобно как временное решение, но плохо подходит для:

- интервального повторения;
- точечной работы с отдельными словами и фразами;
- добавления метаданных на уровень одной карточки;
- последующей автоматической сортировки, фильтрации и обзора прогресса.

Целевая модель должна быть другой: одна заметка на одну языковую единицу.

## Цель

Перевести немецкий навык на структуру, где:

- каждое слово хранится в отдельной markdown-заметке;
- каждая фраза хранится в отдельной markdown-заметке;
- шаблон заметки централизован и может быть изменен после согласования с пользователем;
- учебный план ведется в отдельном файле с учетом статуса выполнения;
- на VDS данные пишутся в `obsidian_vault_simulation`, а затем попадают в мастер-хранилище через транспортный слой;
- логика одинаково работает на VDS и на домашнем сервере.

## Целевая структура папок

Ниже предложена структура внутри Obsidian-корня или симуляции на VDS:

```text
knowledge/
  german/
    words/
      der-tisch.md
      ankommen.md
    phrases/
      wie-geht-es-dir.md
      ich-bin-gerade-unterwegs.md
    grammar/
      passiv-praesens.md
    plans/
      current_learning_plan.md
      archived_plans/
    reviews/
      review_log.md
    _templates/
      word_note_template.md
      phrase_note_template.md
      grammar_note_template.md
      learning_plan_template.md
```

## Основные сущности

### 1. Заметка слова

Одна заметка соответствует одному слову или устойчивой словарной единице.

Согласованный контракт для заметки слова:

- система должна попытаться определить тип слова;
- если это существительное, имя заметки должно быть словом без артикля;
- если это глагол, имя заметки должно быть инфинитивом;
- в поле `wort` для существительного нужно хранить форму с артиклем и окончанием множественного числа;
- в поле `wort` для глагола нужно хранить три формы;
- перевод хранится в `übersetzung`;
- примеры хранятся в `Beispiel`;
- при создании всегда проставляются `created`, `tags`, `status`, `in_dict`.

Шапка заметки слова должна иметь такой вид:

```yaml
---
wort: die Wohnung -en
übersetzung: квартира
Beispiel:
  - Ich suche eine neue Wohnung in Berlin.
  - Die Wohnung ist klein, aber gemütlich.
created: 2026-03-10 21:15
tags:
  - Deutsch
  - Words
status: nova
in_dict: false
---
```

Для глагола:

```yaml
---
wort: gehen, ging, ist gegangen
übersetzung: идти
Beispiel:
  - Ich gehe heute früh nach Hause.
  - Gestern ist er zu Fuß gegangen.
created: 2026-03-10 21:15
tags:
  - Deutsch
  - Words
status: nova
in_dict: false
---
```

Имя файла для существительного должно строиться без артикля, например:

- `die Wohnung -en` -> `wohnung.md`
- `der Tisch -e` -> `tisch.md`

Имя файла для глагола должно строиться по инфинитиву:

- `gehen, ging, ist gegangen` -> `gehen.md`
- `ankommen, kam an, ist angekommen` -> `ankommen.md`

Если тип слова не удалось определить надежно, реализация должна:

1. сохранить заметку как общий word note;
2. использовать нормализованную базовую форму как имя файла;
3. вернуть в статусе, что часть речи определена неуверенно.

### 2. Заметка фразы

Одна заметка соответствует одной фразе, паттерну или готовому выражению.

Для фраз шаблон пока согласован в свободной форме, но рекомендуется такой каркас:

```yaml
---
phrase: Wie geht es dir?
übersetzung: Как у тебя дела?
context: нейтральное повседневное общение
created: 2026-03-10 21:15
tags:
  - Deutsch
  - Phrases
status: nova
in_dict: false
---
```

Дальше в теле заметки:

- раздел `## Usage`;
- раздел `## Examples`;
- раздел `## Notes`.

### 3. Заметка грамматики

Грамматику можно пока оставить как отдельные тематические заметки, а не как одну заметку на правило, если это удобнее. Но интерфейс сохранения должен позволять и поштучный формат.

### 4. Файл учебного плана

Учебный план должен вестись отдельно от лексики и содержать не только сам план, но и учет выполнения.

Для плана формат пока согласован в свободной форме, но рекомендуется такой каркас:

```yaml
---
plan_name: German Learning Plan
created: 2026-03-10 21:15
updated: 2026-03-10 21:15
status: active
tags:
  - Deutsch
  - Plan
---
```

Дальше в теле заметки:

```text
# German Learning Plan

## Goals
- ...

## Current Focus
- ...

## Tasks
- [ ] ...
- [x] ...

## Progress Log
- 2026-03-10: ...

## Review Notes
- ...
```

Если позже окажется, что одного файла мало, можно будет вынести журнал выполнения в отдельный `review_log.md`, но на первом этапе нужен один основной файл как источник истины.

## Требования к реализации

### Функциональные

1. При сохранении слова должен создаваться новый файл, а не дописываться общий `vocabulary.md`.
2. При сохранении фразы должен создаваться новый файл, а не общий список.
3. Если заметка уже существует, система должна:
   - либо обновлять существующую заметку;
   - либо возвращать понятный статус, что запись уже есть;
   - поведение должно быть централизованно определено в одном месте.
4. Должен быть отдельный инструмент для работы с учебным планом:
   - создать план;
   - обновить план;
   - отметить задачу выполненной;
   - добавить запись в лог прогресса.
5. Все записи должны создаваться в VDS-совместимой папке при наличии `obsidian_vault_simulation`.
6. Имена файлов должны быть детерминированными и безопасными.

### Нефункциональные

1. Не ломать существующий транспортный слой.
2. Не завязываться на конкретный шаблон текста до утверждения пользователем.
3. Не терять обратную совместимость там, где это несложно.
4. Код должен быть разделен по ответственности: генерация контента, вычисление путей, запись файла, логика обновления.

## Предлагаемая архитектура изменений

### Новый слой работы с заметками

Нужно выделить отдельный сервис, например:

- `core/skills/german_storage.py`

Ответственность сервиса:

- определять рабочий корень хранилища;
- выбирать режим VDS или Home Server;
- строить пути к словам, фразам, грамматике и планам;
- нормализовать имена файлов;
- создавать заметки по шаблонам;
- обновлять существующие заметки по правилам.

### Обновление навыка учителя

Текущий модуль навыка должен перейти от агрегированных методов к явным операциям:

- `save_word(...)`
- `save_phrase(...)`
- `save_grammar_note(...)`
- `create_or_update_learning_plan(...)`
- `mark_plan_task_done(...)`
- `append_plan_progress(...)`

Метод `save_knowledge(...)` можно временно сохранить как совместимый фасад, но внутри он должен маршрутизировать запись в новые сущности.

### Шаблоны

Шаблоны нужно вынести из кода. На первом этапе достаточно хранить их как markdown-файлы в папке `_templates`.

Важно: word template уже согласован существенно точнее, чем раньше. Для phrase и learning plan допускается рабочая свободная форма, но шаблоны все равно должны быть вынесены из кода.

## Правила именования файлов

Рекомендуется:

- приводить имя к lowercase;
- заменять пробелы на `-`;
- убирать спецсимволы;
- для umlaut и `ß` использовать нормализацию, которая не ломает поиск;
- при коллизии добавлять короткий суффикс.

Примеры:

- `die Wohnung -en` -> `wohnung.md`
- `gehen, ging, ist gegangen` -> `gehen.md`
- `sich kümmern um` -> `sich-kuemmern-um.md`
- `Wie geht es dir?` -> `wie-geht-es-dir.md`

## Правила обновления существующих заметок

Нужна единая стратегия, а не стихийная перезапись.

Базовое поведение:

1. Если файла нет, создать.
2. Если файл есть и пришли новые поля, обновить только соответствующие блоки.
3. Если новые данные конфликтуют со старыми, сохранить оба варианта в секции заметок или вернуть предупреждение.
4. Не затирать пользовательские ручные правки целиком.

## План выполнения

### Этап 1. Уточнение контракта данных

Сделать до начала основной реализации:

1. Зафиксировать сущности: `word`, `phrase`, `grammar`, `plan`.
2. Подготовить шаблон слова по уже согласованному контракту.
3. Подготовить рабочие шаблоны фразы и плана в свободной форме.
4. Реализовать определение типа слова для базовых случаев: существительное, глагол, fallback.

### Этап 2. Реализация файлового сервиса

Сделать:

1. Новый модуль хранения немецких заметок.
2. Функции определения путей для VDS/Home.
3. Слагификацию имен файлов.
4. Базовые операции `create`, `read`, `update-if-exists`.
5. Выделение логики `detect_word_type` и `build_word_filename`.

### Этап 3. Рефакторинг навыка учителя

Сделать:

1. Заменить агрегированную запись в `vocabulary.md` и аналогичные файлы.
2. Подключить новые функции как инструменты для оркестратора.
3. Сохранить понятные статусы результата для LLM и логов.
4. Добавить отдельный путь для существительных и глаголов при сохранении word note.

### Этап 4. Работа с учебным планом

Сделать:

1. Один основной файл плана.
2. Обновление списка задач без полной потери содержимого.
3. Отметку выполненных пунктов.
4. Добавление записей о прогрессе.

### Этап 5. Тестирование

Проверить:

1. Создание новой заметки слова.
2. Создание новой заметки фразы.
3. Повторное сохранение существующего слова.
4. Создание и обновление плана.
5. Работа в режиме `obsidian_vault_simulation`.
6. Проверка, что существительное сохраняется без артикля в имени файла.
7. Проверка, что глагол сохраняется по инфинитиву.
6. Совместимость с экспортом и транспортом.

## Инструкция для другой модели

Ниже инструкция, которую можно дать другой модели для исполнения.

### Роль

Ты реализуешь рефакторинг немецкого навыка в проекте AntigravityAgents. Твоя задача не в том, чтобы улучшить все подряд, а в том, чтобы перевести хранение немецких учебных сущностей на поштучные заметки Obsidian и аккуратный учет плана обучения.

### Что нужно сделать

1. Убрать зависимость немецкого навыка от агрегированных файлов `vocabulary.md`, `grammar.md`, `progress.md` как основной модели хранения.
2. Ввести поштучные заметки для слов и фраз.
3. Ввести один основной файл учебного плана с учетом выполнения.
4. Сохранить совместимость с VDS-режимом через `obsidian_vault_simulation`.
5. Не менять транспортный слой без необходимости.
6. При сохранении слов соблюдать согласованный формат frontmatter и правила имени файла.

### Ограничения

1. Для word note не придумывай новый формат: используй уже согласованный контракт из этого документа.
2. Не удаляй существующие данные автоматически.
3. Не ломай существующие агенты и маршрутизацию в оркестраторе.
4. Не трогай несвязанные части проекта.
5. Для phrase note и learning plan допустима рабочая свободная форма, но шаблоны должны быть вынесены в отдельные файлы.

### Конкретные шаги

1. Проанализируй текущие файлы:
   - `core/utils_obsidian.py`
   - `core/orchestrator_v2.py`
   - `core/skills/german_teacher.py`
2. Создай модуль хранения немецких заметок.
3. Реализуй функцию определения типа слова.
4. Реализуй сохранение слова отдельным файлом с правилами:
  - существительное -> имя файла без артикля;
  - глагол -> имя файла по инфинитиву;
  - `wort` в frontmatter для существительного содержит артикль и множественное число;
  - `wort` в frontmatter для глагола содержит три формы.
5. Реализуй сохранение фразы отдельным файлом.
6. Реализуй операции для учебного плана.
7. Обнови инструменты оркестратора так, чтобы они вызывали новые методы.
8. Добавь локальные тесты на файловый результат в `obsidian_vault_simulation`.

### Критерии приемки

Считать задачу выполненной можно только если:

1. Новое слово создает отдельную markdown-заметку.
2. Новая фраза создает отдельную markdown-заметку.
3. Повторная запись не уничтожает ручное содержимое заметки целиком.
4. Учебный план ведется отдельным файлом и поддерживает отметку выполнения.
5. В режиме VDS все результаты появляются в `obsidian_vault_simulation` и подхватываются транспортом.
6. Для существительного имя файла не содержит артикль.
7. Для глагола имя файла соответствует инфинитиву.
8. Frontmatter word note содержит `wort`, `übersetzung`, `Beispiel`, `created`, `tags`, `status`, `in_dict`.

### Что не делать сейчас

1. Не внедрять полноценный алгоритм интервального повторения на этом шаге.
2. Не менять формат данных в транспортном слое.
3. Не проектировать сложную базу данных для немецкого навыка.

## Следующее решение, которое нужно согласовать с пользователем

До кодирования финальной схемы заметок нужно отдельно согласовать шаблоны:

1. правила определения части речи в неоднозначных случаях;
2. правила обновления уже существующей заметки;
3. формат секции примеров и заметок внутри phrase note;
4. формат секции задач и прогресса внутри learning plan.
