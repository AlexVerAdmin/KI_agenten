# 🧠 Концепция Долговременной Памяти (Memory)

## Проблема
Агенты (и я, Copilot) имеют "окно контекста". Когда оно забивается кодом, мы забываем:
- Глобальные цели (например, почему мы выбрали именно Obsidian).
- Твой текущий статус (какой прогресс в обучении).
- Принятые архитектурные решения.

## Решение: Трёхуровневая Память

### 1. Оперативная (Short-term)
- **Что это:** Последние 20-30 сообщений в чате.
- **Хранение:** SQLite `chat_history`.
- **Проблема:** Быстро "вымывается" новыми сообщениями.

### 2. Рабочая (Mid-term / Working Context)
- **Что это:** Файл `current_session.md` или `active_context.md`.
- **Как работает:** В конце каждого логического блока (или раз в N сообщений) агент вызывает `skill_summarize()`.
- **Результат:** Краткая выжимка: "Что сделали", "На чем остановились", "Какие следующие шаги". Этот файл **всегда** подгружается в System Prompt.

### 3. Фундаментальная (Long-term / Knowledge)
- **Что это:** Твой `student_profile.md`, `learning_plan.md`, `project_structure.md`.
- **Как работает:** Агент обновляет эти файлы только когда происходит качественное изменение (выучил тему B2, сменил архитектуру базы).

---

## 🛠 Предлагаемая структура Скилла Памяти:

### `skill_commit_memory(summary_text)`
1. Берет текущий `active_context.md`.
2. Добавляет/обновляет в нем информацию на основе `summary_text`.
3. Очищает (по возможности) историю чата, если контекст уже перегружен, оставляя только выжимку.

## Что это даст Herr Max Klein?
Когда ты начнешь урок, он:
1. Прочитает **Профиль** (Кто ты).
2. Прочитает **План** (Что учим).
3. Прочитает **Active Context** ("Вчера мы обсуждали интервью, сегодня продолжаем с того же места").

**Как тебе такая структура?** Если одобряешь, я начну создавать `active_context.md` и универсальный скилл для управления им.
