# План Реализации Transport Layer Для Памяти И Агентов

## Назначение Файла
Этот файл написан как пошаговое ТЗ для младшей модели.
Цель: реализовать транспорт данных между Home Server и VDS без GitHub для runtime-данных.

Использовать этот файл как источник правды:
- при разработке здесь, на локальной машине;
- при передаче задачи на VDS для тестирования и исправления ошибок;
- при дальнейшем подключении немецкого учителя и консультанта по трудоустройству.

---

## Главная Идея
GitHub остается только для кода.

Runtime-данные не передаются через GitHub:
- SQLite snapshots;
- summary-файлы памяти;
- служебные state-файлы;
- агентские промежуточные результаты.

Для runtime-данных использовать прямой транспорт по SSH внутри сети/между узлами.

Базовая схема:
1. VDS экспортирует snapshot или batch новых записей.
2. Home Server забирает данные по SSH.
3. Home Server запускает локальную модель для суммаризации.
4. Home Server отправляет summary обратно на VDS по SSH.
5. VDS применяет summary в файлы памяти и знаний.

## Участники Системы

### 1. Copilot Memory
Использует summary для поддержания рабочего контекста в VS Code.

### 2. Herr Max Klein
Немецкий учитель на VDS.
Должен получать обновления памяти и словаря после обработки на Home Server.

### 3. Career Consultant
Консультант по трудоустройству.
Тоже должен участвовать в архитектуре транспорта.
Он будет читать и получать свои summary-файлы отдельно от немецкого учителя.

### 4. Home Server
Запускает локальную младшую модель или локальную summarization-модель.
Отвечает за обработку истории и генерацию выжимок.

### 5. VDS
Источник боевых данных и место применения результатов.

---

## Ограничения И Правила

### Безопасность
1. Никогда не отправлять секреты в Markdown summary.
2. Никогда не публиковать runtime-данные в GitHub.
3. Любые токены, ключи, пароли маскировать до записи результата.

### Надежность
1. Не читать боевую SQLite напрямую во время записи, если можно сделать snapshot.
2. Не переписывать файлы памяти целиком без необходимости.
3. Любая операция транспорта должна быть идемпотентной.

### Разделение Контекста
Нельзя смешивать в одном summary:
- Copilot;
- German Teacher;
- Career Consultant;
- общую системную инфраструктуру.

## Фаза 1. Реализация Здесь, На Home/Repo

Цель фазы: подготовить код и локальный dry-run без обязательной привязки к VDS.

### Что делает младшая модель
1. Создает папку `core/transport/`.
2. Создает скрипты-заготовки.
3. Проверяет локальный сценарий на тестовых файлах.
4. Не лезет в боевой деплой на VDS до завершения локального dry-run.

### Файлы, которые нужно создать
1. `core/transport/common.py`
   Назначение:
   - общие пути;
   - загрузка env;
   - работа со state-файлами;
   - маскирование секретов.

2. `core/transport/pull_snapshot_from_vds.py`
   Назначение:
   - забирать snapshot или batch-файл с VDS по SSH/SCP.

3. `core/transport/push_summary_to_vds.py`
   Назначение:
   - отправлять готовые summary-файлы обратно на VDS.

4. `core/transport/process_incoming_batch.py`
   Назначение:
   - выбирать входящий batch;
   - запускать summarizer;
   - сохранять результат в `runtime_sync/outgoing/...`.

5. `core/transport/apply_summary_local_test.py`
   Назначение:
   - локально тестировать применение summary к Markdown-файлам.

### Что пока не делать
1. Не подключать сразу cron/systemd.
2. Не делать сразу delta-sync по `last_id`, если snapshot-путь еще не работает.
3. Не объединять transport с skills в одном файле.

### Локальный dry-run
Нужно прогнать 3 сценария:

1. `copilot`
   - вход: тестовый batch истории;
   - выход: summary в память Copilot.

2. `german`
   - вход: тестовый batch учителя;
   - выход: summary или обновление плана/вокабуляра.

3. `career`
   - вход: тестовый batch консультанта по трудоустройству;
   - выход: summary в отдельный файл карьерного трека.

### Критерии Готовности Фазы 1
1. Все 5 файлов созданы.
2. Есть локальная симуляция входящего batch.
3. Есть локальный summary-результат по трем трекам: `copilot`, `german`, `career`.
4. Секреты маскируются.
5. State-файлы обновляются корректно.

## Фаза 2. Передача На VDS И Тестирование

Цель фазы: запустить реальный транспорт между узлами.

### Что делает младшая модель
1. Переносит `core/transport/` и `runtime_sync/` на VDS.
2. Настраивает доступ по SSH.
3. Проверяет, что VDS умеет экспортировать данные, а Home Server умеет их забирать.

### Что должно появиться на VDS
1. Скрипт экспорта snapshot или batch-файла.
2. Папка `runtime_sync/export/...`.
3. Папка `runtime_sync/import/...`.
4. State-файлы для каждого трека.

### Тестовые сценарии

#### Сценарий A. Copilot
1. На VDS появляется новый batch истории.
2. Home Server его забирает.
3. Локальная модель создает summary.
4. Summary возвращается на VDS.
5. VDS обновляет `active_memory.md` или соответствующий файл.

#### Сценарий B. German
1. Учитель создает новые записи.
2. Они попадают в transport batch.
3. Home Server делает выжимку.
4. Результат возвращается в немецкий knowledge-track.

#### Сценарий C. Career
1. Консультант по трудоустройству создает/обновляет свой контекст.
2. История уходит в отдельный career-batch.
3. Home Server суммаризирует.
4. Результат возвращается в career summary/plan файлы.

### Сценарий D. Obsidian & Copilot Sync (NEW)
1. **Copilot History**: Инкрементальный экспорт `copilot_history.md` (только новые блоки).
2. **Obsidian Vault Simulation**: Поиск новых/измененных `.md` файлов на VDS.
3. **Sync**: Дальнейшая обработка на Home Server для вливания в Master Vault и генерации связей.

### Критерии Готовности Фазы 2
1. Реальный SSH transport работает туда и обратно.
2. Summary-файлы доходят на VDS.
3. Summary применяется без ручного редактирования.
4. Ошибка в одном треке не ломает другие.

## Фаза 3. Инкрементальный Режим

Цель фазы: уменьшить объем передачи и ускорить цикл.

### Что делает младшая модель
1. Добавляет `last_exported_id` и `last_processed_id`.
2. Переходит от snapshot к incremental batch export.
3. Добавляет повторную обработку при падении без дублей.

### Нужно реализовать
1. Экспорт только новых записей.
2. Раздельные state-файлы для `copilot`, `german`, `career`, `infra`.
3. Защиту от повторного применения одного и того же summary.

### Критерии Готовности Фазы 3
1. Передаются только новые данные.
2. Нет дублей в памяти.
3. Повторный запуск безопасен.

## Фаза 4. Интеграция С Архитектурой Skills

Цель фазы: transport становится частью общей skill/service-архитектуры.

### Что делает младшая модель
1. Переносит повторяющиеся функции в `core/skills/` или сервисный слой.
2. Подключает transport к учителю и карьерному консультанту через единый интерфейс.
3. Разделяет:
   - transport;
   - summarize;
   - apply-to-files;
   - agent-specific logic.

### Для Herr Max Klein
1. Подключить transport к vocabulary и plan update pipeline.
2. Затем добавить интервальное повторение.

### Для Career Consultant
1. Подключить transport к файлам карьерного плана.
2. Отдельно суммаризировать карьерные решения, вакансии, CV notes и interview prep.

7. Локально протестировать `german`.
8. Локально протестировать `career`.
9. Только после этого переносить код на VDS.
10. На VDS тестировать по одному треку, начиная с `copilot`.

## Ясные Правила Для Младшей Модели

1. Не использовать GitHub для runtime-файлов.
2. Не смешивать German и Career в один summary.
3. Не переписывать большие Markdown-файлы без причины.
4. Не трогать боевые промпты агентов, пока transport не стабилен.
5. Любую ошибку сначала воспроизводить локально, потом исправлять, потом переносить на VDS.

---

## Минимальный Результат, Который Считается Успехом

Успех этапа transport считается достигнутым, если:
1. Home Server автоматически забирает batch с VDS.
2. Локальная модель суммаризирует его.
3. Summary возвращается на VDS.
4. Copilot, Herr Max Klein и Career Consultant получают свои файлы памяти отдельно.
